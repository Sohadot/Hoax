# Non-Public Static Workbench Visual System Hardening v1

## Purpose

This document governs the first visual system hardening of the internal static Evidence Posture Workbench prototype. It strengthens visual grammar, semantic support, spatial hierarchy, and Hoax-specific evidence chamber identity while keeping the prototype static, non-public, and non-operational.

## Non-Purpose

This sprint does not:

- create new prototype files;
- create a new prototype directory;
- create a public workbench;
- create a public route;
- create a sitemap entry;
- create public navigation;
- create an engine;
- create a classifier;
- create a detector;
- create upload functionality;
- create scoring;
- create fake/real outputs;
- create forms;
- create user inputs;
- create analytics;
- create an API;
- create monetization;
- modify DNS or Cloudflare;
- launch the Hoax.ai custom domain;
- authorize deployment.

## Governing Principle

Visual hardening may strengthen conceptual identity. It must not expand capability.

The Hoax.ai visual system must make evidence posture legible without becoming a detector aesthetic.

## Visual System Hardening Definition

Visual system hardening is a controlled strengthening of the internal static prototype's visual grammar, CSS tokens, conceptual patterns, spatial hierarchy, accessibility support, responsive stability, and Hoax-specific evidence chamber identity without adding files, functionality, routes, sitemap entries, public links, JavaScript, forms, inputs, upload behavior, scoring, engine behavior, classifier behavior, API behavior, analytics, or deployment changes.

## Allowed Scope

Allowed files:

- _internal_prototypes/evidence-posture-workbench/index.html
- _internal_prototypes/evidence-posture-workbench/prototype.css

Forbidden:

- any additional prototype file;
- any additional prototype directory;
- any public page;
- any public route;
- any sitemap expansion;
- any public navigation link.

The hardening must improve:

1. Evidence field background system
2. Evidence chamber frame
3. Artifact-subject boundary rails

5. Missing context absence fields
6. Not-assessable protected state
7. Refusal gate visual hierarchy
8. Output envelope containment
9. Verification path legibility
10. Static state rhythm
11. Mobile/responsive coherence

13. Anti-detector differentiation
14. Anti-upload dashboard differentiation
15. Anti-scoring dashboard differentiation
16. Anti-SaaS dashboard differentiation

## Visual Token Direction

The CSS may introduce or strengthen internal custom properties for background field, chamber surface, chamber edge, graphite text, muted text, boundary accent, provenance line, provenance shadow, absence field, not-assessable surface, refusal surface, output envelope surface, verification path line, subtle rule, focus ring, and mobile spacing.

Rules:

- Tokens must live inside prototype.css only.
- Tokens must not import external systems.
- Tokens must not create dependency on external fonts, URLs, libraries, scripts, images, APIs, or assets.
- Tokens must express Hoax.ai's governed evidence field direction, not generic dashboard theming.

## Conceptual Visual Primitives

1. Evidence Field: The page background should feel like an evidence condition field, not decoration.
2. Evidence Chamber: The central workspace should feel contained, governed, and institutionally calm.
3. Boundary Rail: Artifact-subject separation should be visually legible as a boundary, not a generic divider.
4. Provenance Shadow: Missing origin, source, or custody must feel like confidence-limiting absence, not proof of deception.
5. Missing Context Absence: Missing information should appear as a structural condition.
6. Not-Assessable Restraint: Not Assessable should appear protected and disciplined, not as error or failure.
7. Refusal Gate: Refusal should appear as governance, not blocked frustration.
8. Output Envelope: Output should appear contained, bounded, and non-verdict.
9. Verification Path: Questions should appear as inquiry paths, not proof of verification.
