# Hoax.ai Interface Embodiment Governance

**Version:** v1.0.0  
**Status:** governed_internal_interface_standard  
**Maturity:** not_public_tool  
**Decision:** DEC-023

## A. Purpose

Hoax.ai's interface must embody the thesis of evidence posture and make governance boundaries visible.

The interface is not decoration. It is a governed explanation layer that helps users understand evidence structure, uncertainty, and boundaries — without implying detection authority, verdict output, or active tool capability.

## B. Non-Purpose

Interface governance does **not**:

- create a public tool;
- create a public classifier;
- create upload functionality;
- create scoring;
- create a scanner;
- create a fake/real detector;
- create a surveillance dashboard;
- create public deployment readiness;
- create new routes;
- create SEO expansion.

## C. Governing Principle

**The interface must embody evidence posture, not simulate detection authority.**

**A Hoax.ai interface may show structure, posture, uncertainty, source conditions, and boundaries; it must not visually imply verdict, certainty, accusation, or automated detection.**

## D. Interface Thesis

The interface must explain the condition of evidence through structure:

- artifact boundary
- source chain
- provenance gap
- contextual stability
- coherence questions
- evidence chain continuity
- subject separation
- output boundary

## E. Allowed Visual Metaphors

- layered evidence surface
- document strata
- evidence chain
- bounded frame
- posture map
- source path
- uncertainty field
- separated artifact/subject zones
- governance rail
- decision boundary
- inspection ledger
- muted institutional signal layer

## F. Prohibited Visual Metaphors

- scanner beam
- red threat dashboard
- fake/real switch
- truth meter
- lie detector
- police evidence board
- surveillance grid
- crosshair targeting
- biometric targeting
- cyber-gaming detection UI
- alarm-heavy fraud console
- AI magic detector
- upload-to-detect panel
- subject risk profile
- guilt meter
- authenticity score gauge

## G. Interface Component Rules

Every component must have: component_id, name, purpose, thesis_function, governance_boundary, allowed_use, prohibited_use, public_status, dependency, notes.

No component may exist only for decoration.

## H. Current Public Surface Rule

The current `index.html` is a thesis surface only.

It may explain: what Hoax.ai is; what it refuses to do; evidence artifact boundary; artifact-subject separation; self-application; governance posture.

It must not imply: active classifier; upload workflow; detection tool; scoring; fake/real verdict; public reports; API access; enterprise service availability.

## I. Future Interaction Rule

Future interaction is blocked until:

- output boundary schema exists;
- internal engine model exists;
- adversarial harness passes;
- interface component is registered;
- security/privacy boundary exists;
- deployment readiness gate passes.

## J. Accessibility and Performance Rule

The interface must remain:

- readable without JavaScript;
- mobile-safe;
- static-first;
- accessible by semantic structure;
- restrained in motion;
- not dependent on external libraries without approval;
- not dependent on WebGL or canvas for core meaning.
