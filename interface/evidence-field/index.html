Hoax.ai Evidence Field Interface Thesis

The Hoax.ai Evidence Field Interface is a future interface model for visualizing evidence posture, protocol sequence, evidence limitation, interpretation risk, and output boundaries without converting review into detection, scoring, accusation, or automated verdict.

Interface Thesis Statement

This page is a public interface thesis—a visual philosophy and design foundation derived from Evidence Posture Standard v1 and Evidence Posture Protocol v1 Draft. It defines how Hoax.ai may visualize evidence conditions without behaving like a detector dashboard, scanner, classifier, upload interface, scoring system, API front end, moderation console, public report generator, or automated verdict interface.

The interface thesis embodies evidence posture. It does not decorate Hoax.ai with generic AI-product aesthetics. Future interface embodiment—if authorized under separate gates—must make boundaries, uncertainty, and output restraint legible rather than hiding them behind single-result dominance.

Why Interface Matters

Hoax.ai cannot use a normal AI-detector dashboard because detector conventions encode the wrong epistemology. Detector dashboards imply binary outcomes: content is treated as passing or failing a hidden test. Upload boxes imply automated classification—users submit material and expect a machine judgment. Numeric scores imply false precision where evidence envelopes are actually qualified, limited, or not assessable. Result cards imply verdict behavior: one dominant panel tells the reader what to believe. Red and green user-interface patterns imply truth-or-falsity judgment rather than artifact condition under boundary rules.

Institutional buyers evaluating Hoax.ai should ask whether a proposed interface visualizes boundaries or hides them. If the first screen suggests “submit and receive judgment,” the asset has already been misread as a tool. If the first screen suggests “review artifact condition under standard and protocol,” the interface thesis aligns with repository substance. Sprint 63 publishes that distinction as public doctrine so future design sprints inherit the correct metaphor.

Journalists, analysts, trust and safety teams, and compliance reviewers need interfaces that document limitation and refusal—not interfaces that compress review into a single badge. Why interface matters is therefore a governance question, not a styling preference.

Hoax.ai needs boundary-aware visualization, not verdict display. Institutions reading Hoax.ai need to see support conditions, provenance gaps, context collapse, claim drift, interpretation risk, and attribution boundaries as first-class visual facts—not as footnotes after a headline verdict. A standard-bearing evidence-posture asset that adopted detector-dashboard conventions would contradict its own reference layer, standard, and protocol.

Interface form is not neutral. If the first public visual language looks like upload-and-scan, every subsequent layer inherits that misread. Sprint 63 establishes interface thesis before any operational surface so future design cannot drift into tool behavior by accident. Institutional reviewers should treat any future mock that centers an upload box or scan animation as a governance violation until explicitly re-authorized.

Evidence Field, Not Detector Dashboard

Evidence Field means evidence conditions are spatially related within one governed view. Claims, sources, provenance, context, traceability, limitation, interpretation risk, and output boundaries coexist. Outputs are bounded by support. Uncertainty is visible. Boundaries are visible. No single “result” dominates the page. The reader walks relationships among artifact, claim, and source basis—not a funnel toward one badge.

Detector Dashboard means upload-centered layout, scan metaphor, classify workflow, traffic-light verdict presentation, confidence meter, and headline certainty. Hoax.ai must reject detector-dashboard framing explicitly. This thesis page is evidence-field and protocol-visualization doctrine—not a feature announcement for a tool that does not exist on this surface.

Repository file PROTOCOL_TO_INTERFACE_READINESS.md documents internal readiness constraints. This public thesis makes those constraints legible to buyers, designers, and institutional reviewers evaluating Hoax.ai as a protocol-bearing asset.

Interface Layers

The future interface is layered, not linear. Each layer maps to reference concepts and protocol steps without collapsing into one verdict column.

  • Artifact Layer — structural observability; EP-P01
  • Claim Layer — claim text and movement; EP-P02, EP-P11
  • Source Basis Layer — cited sources; EP-P03, EP-P04
  • Provenance Layer — custody and gaps; EP-P05
  • Context Layer — situational frames; EP-P06, EP-P10
  • Traceability Layer — claim-to-source mapping; EP-P07
  • Evidence Limitation Layer — support envelope; EP-P12
  • Interpretation Risk Layer — overreading restraint; EP-P13
  • Attribution Boundary Layer — artifact-only scope; EP-P14
  • Output Boundary Layer — language refusal; EP-P15, EP-P17
  • Posture State Layer — Supported, Qualified, Limited, Not Assessable, Out of Scope; EP-P16

Layers may be shown as adjacent regions in an evidence-field map. They must not stack into a single “detection result” card. In future static embodiment, each layer should remain inspectable independently: collapsing provenance into posture without showing the gap is a visual form of gap-to-manipulation conversion. Collapsing attribution boundary into artifact description is a visual form of artifact-to-subject transfer. The interface thesis requires layer integrity even when screen space is limited.

Posture State Layer displays categorical labels from the standard—never numeric ranks. Supported does not mean true; Not Assessable does not mean false. Visual treatment must reinforce that discipline with typography and hierarchy, not with traffic-light color logic.

Evidence Field Static Embodiment v1

Sprint 64 adds a static, accessible visual structure that embodies the interface thesis below. This is semantic HTML and CSS only—no JavaScript, no forms, no inputs, no upload areas, no scan controls, no score meters, and no result cards. Each zone represents an evidence condition or boundary, not an automated output. Read top to bottom; zones are related conditions, not steps toward a verdict.

If styles are unavailable, the ordered list below still conveys the full evidence-field structure. No color, animation, or layout is required for comprehension.

  1. Artifact Layer

    Observed object under review—format, structure, and observable properties. Artifact condition is described; no subject guilt or intent is assigned. This layer anchors review in what can be seen on the artifact, not who produced it.

  2. Claim Layer

    Statement attached to or derived from the artifact. Claims may drift across circulation; traceability to source basis remains required. A claim is evaluated as text-in-context, not as a truth verdict.

  3. Source Basis Layer

    Available source records within scope. Source confidence describes chain posture—not truth certification. Weak source basis limits how strongly any posture may be stated.

  4. Provenance Layer

    Custody path or documented gap. Provenance gaps limit interpretation but do not prove manipulation. A gap is a boundary fact, not an accusation.

  5. Context Layer

    Situational frames affecting interpretive reach. Missing context limits posture; context collapse must be visible before stronger language appears.

  6. Traceability Layer

    Claim-to-source linkage integrity from Claim–Source Traceability. Weak traceability limits output strength; the interface shows the link, not a fact-check authority.

  7. Evidence Limitation Layer

    Support envelope from Evidence Limitation. Shows what can be said, what must be qualified, and what cannot be decided from available evidence.

  8. Interpretation Risk Layer

    Overreading danger from Interpretation Risk. Risk is a restraint signal—not a verdict about truth or falsity.

  9. Attribution Boundary Layer

    Barrier preventing artifact condition from becoming subject accusation. Governed by Attribution Boundary and Artifact–Subject Separation. This layer is visually and textually prominent because subject-transfer is the highest-risk interface failure mode.

  10. Output Boundary Layer

    Allowed and prohibited output language from Output Boundary. Output restraint is part of the interface—not an afterthought panel.

  11. Evidence Posture Layer

    Categorical posture states only—no numeric ranks:

    • Supported — evidence supports stated artifact condition within boundaries
    • Qualified — support exists with explicit caveats
    • Limited — narrow support envelope; strong claims refused
    • Not Assessable — insufficient basis for posture assignment
    • Out of Scope — review outside governed artifact scope

    Supported does not mean true. Not Assessable does not mean false. No traffic-light presentation. No headline “result” dominates this field.

Protocol Path

The following path maps the static embodiment to Evidence Posture Protocol v1 Draft steps EP-P01 through EP-P17. It is a non-scoring progression showing review sequence—not automated execution.

This path is explanatory and non-operational. It does not execute review, produce a result, classify an artifact, or generate a score. No public protocol runner exists on this surface.

  1. EP-P01 Identify the artifact
  2. EP-P02 Identify the claim
  3. EP-P03 Locate the source basis
  4. EP-P04 Assess source confidence
  5. EP-P05 Review provenance continuity
  6. EP-P06 Review context completeness
  7. EP-P07 Check claim-source traceability
  8. EP-P08 Review evidence chain continuity
  9. EP-P09 Check synthetic fragility
  10. EP-P10 Check context collapse
  11. EP-P11 Check claim drift
  12. EP-P12 Define evidence limitation
  13. EP-P13 Assess interpretation risk
  14. EP-P14 Apply attribution boundary
  15. EP-P15 Apply output boundary
  16. EP-P16 Assign evidence posture state
  17. EP-P17 Produce bounded language only

Each step relates to EPS principles and reference concepts. Visualization documents sequence for institutional audit; the interface does not walk the path on behalf of the reviewer.

Boundary Rail

The boundary rail makes allowed and prohibited inference visible before any output language forms. It is a static guardrail—not an interactive control.

Allowed

  • evidence condition
  • limited posture
  • source caveat
  • provenance caveat
  • context caveat
  • traceability caveat
  • interpretation risk caveat
  • attribution boundary caveat
  • bounded output language

Prohibited

  • fake/real verdict
  • subject guilt
  • deception intent
  • source fraud claim
  • manipulation proof
  • responsibility assignment
  • score
  • automated result
  • upload classification

The terms fake and real appear here only as prohibited-language examples—not as interface states, labels, or posture values.

Reading the Field

Read the static embodiment from top to bottom. Each zone is a condition under review—not a step toward a single outcome. Layers coexist; none should dominate as a “detection result.”

  • Zones represent artifact, claim, source, provenance, context, traceability, limitation, interpretation risk, attribution boundary, output boundary, and posture—together, not as a funnel.
  • Boundaries constrain what may be said; they are not decorative borders.
  • Posture state emerges conceptually from reviewed conditions—it is not auto-generated by this page.
  • No result is generated here. This is evidence-field and protocol-visualization doctrine embodied in static HTML.
  • Reject upload-centered layout, scan metaphor, traffic-light verdict presentation, confidence-score meter, and result-card framing.

Protocol Visualization

Evidence Posture Protocol v1 Draft defines EP-P01 through EP-P17 as a review sequence path. A future interface may visualize that path as a Protocol Path—a non-scoring progression where each step relates to EPS principles and reference concepts. Visualization is documentation of sequence, not automated execution. No public protocol runner exists on this surface.

Protocol visualization must show step relationships without numeric ranking, without percentage confidence for public ordering, and without classifier-style labels presented as verdicts. Steps may appear complete or incomplete in institutional review records; they must not auto-advance based on hidden models. Human-readable protocol visualization is authorized in principle; public engine behavior is not.

Each EP-P step links to hardened reference pages. Interface panels—if ever authorized—should cite those links rather than invent parallel vocabulary. EP-P01 and EP-P02 establish scope before any posture display. EP-P12 through EP-P14 must appear before EP-P16 in visual flow so limitation and boundaries precede posture labels. EP-P17 remains terminal: no “preview result” panel may appear before output boundary review is represented.

Protocol visualization is documentation of sequence for institutional audit—not a progress bar toward verdict. Steps may be marked complete in internal records; the public thesis does not implement completion tracking. Reviewers walk the path; the interface does not walk it for them.

Standard Alignment

The static embodiment must obey Evidence Posture Standard v1. Selected EPS principles govern how each zone may be read:

  • EPS-001 — evidence condition before conclusion; zones precede any posture label
  • EPS-002 — artifact condition is not subject guilt; Artifact and Attribution Boundary layers enforce this
  • EPS-006 — output must remain inside support boundaries; Output Boundary and Evidence Limitation layers make this visible
  • EPS-012 — evidence limitation defines responsible speech; limitation is a first-class zone
  • EPS-013 — interpretation risk requires output restraint; risk is visible, not hidden
  • EPS-014 — attribution boundary prevents subject-transfer overclaim; boundary rail reinforces refusal

Allowed language boundaries and prohibited language boundaries must remain visible—not buried. Posture states and support conditions from the standard govern the Evidence Posture Layer. The interface must not convert Supported into truth certification, Limited into weakness accusation, or Not Assessable into hidden false judgment.

No verdict conversion. No subject accusation. Standard Alignment Markers in a future interactive layer would indicate which EPS principles constrain each view; this static embodiment demonstrates the requirement without implementing markers.

Accessibility and Trust

The interface must remain readable without motion. It must work as static HTML first—this thesis page demonstrates that constraint. No meaning may be hidden only in color. No animation is required for comprehension. Decorative complexity that weakens institutional trust is prohibited.

Accessible fallback must exist before advanced visual layers. Any future diagram must duplicate its meaning in text, as the Evidence Field Map text diagram does here. Trust in Hoax.ai as a standard-bearing asset depends on clarity, not spectacle.
