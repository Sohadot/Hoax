# Non-Public Static Workbench Prototype Governance

**Version:** v1.0.0  
**Status:** governed_non_public_static_workbench_prototype_governance  
**Maturity:** prototype_governance_only_no_prototype_no_interface_no_engine_no_classifier_no_tool  
**Decision:** DEC-051

## A. Purpose

This document governs a future non-public static prototype for the Evidence Posture Workbench.

## B. Non-Purpose

This sprint does **not** create prototype files, a prototype directory, a public workbench, non-public prototype interface, HTML/CSS/JS implementation, public engine, public classifier, public tool, upload functionality, scoring, fake/real outputs, forms, analytics, API, monetization, public routes, sitemap expansion, DNS/Cloudflare changes, custom domain launch, or deployment authorization.

## C. Governing Principle

**Prototype governance defines what may be built later. It must not build it now.**

**A non-public prototype may explore interface form, but it must not create operational capability.**

## D. Meaning of Non-Public

Non-public means: not linked from the public site, not included in sitemap.xml, not registered as a public route, not exposed as a public product, not presented as a workbench launch, and not treated as an engine or classifier.

Non-public does not mean secret if the repository is visible. It means non-public as a web surface and non-operational as a system.

## E. Future Prototype Definition

The future prototype is a static, non-operational, repository-visible but non-public web-surface prototype that may visually explore the Evidence Posture Workbench's evidence chamber layout, zones, state displays, refusal displays, output envelope containment, and evidence field background without processing data, accepting input, uploading files, generating results, scoring, classifying, storing data, calling APIs, or creating public routes.

## F. Future Prototype Location

The only allowed future prototype location:

**`_internal_prototypes/evidence-posture-workbench/`**

- The location is future-allowed only.
- Do not create this directory in Sprint 33.
- The future prototype must not be added to sitemap.xml.
- The future prototype must not be added to data/route-registry.json as a public route.
- The future prototype must not be linked from index.html or public reference pages.
- The future prototype must remain out of public navigation.
- The future prototype must not require .nojekyll.

## G. Static-Only Rule

The future prototype may eventually include static HTML and CSS only, unless later governance explicitly permits otherwise.

Default restrictions: no JavaScript, no forms, no inputs, no upload controls, no network calls, no storage, no analytics, no API calls, no generated outputs, no interactive classification, no scoring behavior, no fake/real behavior.

## H. Visual Identity Requirement

The future prototype must follow the validated conceptual direction: evidence chamber not detector; governed evidence field not black cyber dashboard; artifact first; boundary first; posture before verdict; provenance shadow; missing context as absence; not assessable as protected restraint; refusal as governance; output envelope containment; no verdict gravity.

The future prototype background must not default to generic black cyber UI. It should explore the governed evidence field direction validated in Sprint 32.

## I. Future Prototype Allowed Visual Explorations

The future prototype may visually explore: evidence chamber frame; artifact-subject boundary rail; evidence field background; provenance shadow layer; missing information register; not-assessable lock; refusal gate; output envelope box; verification path; static state map; static component layout.

## J. Future Prototype Forbidden Visual Patterns

The future prototype must not use: detector dashboard; scanner UI; upload box as central element; red/green fake-real result logic; truth meter; risk score; probability gauge; forensic game visuals; police/control-room aggression; SaaS analytics dashboard; product landing-page CTA layout; "try it now" pattern.

## K. Future Prototype Content Rules

The future prototype may use fictional placeholder labels only. It must not include real people, real companies, real institutions, real brands, political events, current events, accusations, external factual claims, uploaded media, URLs to evidence, generated user-facing analysis, fake/real outputs, scores, verification claims, or certification claims.

## L. Future Prototype Review Gates

Before any future prototype file is created, these gates must pass:

1. Prototype location gate
2. Non-public route gate
3. Static-only gate
4. No input/upload gate
5. No scoring/verdict gate
6. No engine/classifier gate
7. Evidence field visual identity gate
8. Accessibility/performance gate
9. No public navigation gate
10. No sitemap expansion gate
11. No deployment/DNS/Cloudflare gate
