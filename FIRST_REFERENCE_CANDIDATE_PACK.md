# Hoax.ai First Reference Candidate Pack

**Version:** v1.0.0  
**Status:** governed_internal_candidate_pack  
**Maturity:** candidates_only_no_drafts_no_routes_no_publication  
**Decision:** DEC-035

## A. Purpose

This document introduces the first internal reference candidates for future Hoax.ai reference expansion.

## B. Non-Purpose

This candidate pack does **not**:

- create public pages;
- create draft pages;
- create route files;
- activate routes;
- expand sitemap.xml;
- create SEO expansion;
- create structured data on public pages;
- create public classifier functionality;
- create a public tool;
- create upload functionality;
- create scoring;
- create forms;
- create analytics;
- deploy anything;
- enable DNS or Cloudflare;
- authorize publication.

## C. Governing Principle

**A candidate is not a page. It is a governed permission request for future reference value.**

**The first candidate pack must prove discipline before it proves scale.**

## D. Candidate Definition

A reference candidate is an internal governed record proposing a future reference unit. It includes a reference purpose, page family, concept scope, claim scope, source scope, semantic SEO role, prohibited misreadings, required gates, and blocked public status.

## E. Candidate Status

All Sprint 17 candidates are:

- internal_only
- candidate_registered
- not_route
- not_page
- not_draft
- not_sitemap_eligible
- not_public
- publication_blocked

## F. Candidate Pack Scope

This first pack contains eight foundational candidates from the governed Hoax.ai system:

1. Evidence Posture (REF-CAND-0001)
2. Artifact–Subject Separation (REF-CAND-0002)
3. Source Confidence (REF-CAND-0003)
4. Provenance Gap (REF-CAND-0004)
5. Not Assessable Posture (REF-CAND-0005)
6. Output Boundary (REF-CAND-0006)
7. Claim and Source Traceability (REF-CAND-0007)
8. Synthetic Fragility (REF-CAND-0008)

No route directories, HTML pages, markdown drafts, sitemap entries, public navigation links, public metadata, or external factual claims were created.

## G. Required Candidate Fields

Each candidate defines: candidate_id, candidate_name, candidate_status, page_family, page_type_ref, candidate_slug, proposed_path, route_status, sitemap_status, draft_status, publication_status, reference_thesis, reference_purpose, definition_scope_summary, governance_boundary_summary, system_relationships, claim_scope, source_scope, semantic_seo_role, prohibited_misreadings, required_sections, required_gates, related_existing_governance_files, future_internal_link_targets, forbidden_public_implications, review_status, and notes.

## H. Candidate Gate Requirements

Each candidate requires: reference blueprint, content quality, claim/source traceability, source scope, semantic SEO governance, route/link integrity, technical quality, interface governance, security/privacy, forbidden language, publisher dry-run, human/governance approval, validate_all.py, and audit record gates.
