## A. Purpose

This document defines the non-operational interface blueprint governance for the future Evidence Posture Workbench.

## B. Non-Purpose

This sprint does **not** create a public workbench, non-public prototype interface, public engine, public classifier, public tool, upload functionality, scoring, fake/real outputs, forms, analytics, API, monetization, public routes, sitemap expansion, DNS/Cloudflare changes, custom domain launch, or deployment authorization.

## C. Governing Principle

**An interface blueprint may define the shape of interaction. It must not become an interface.**

**The future workbench interface must embody evidence posture reasoning, not simulate detection authority.**

**The interface must make Hoax.ai's thesis visible: evidence should be structured before it is believed, escalated, published, or judged.**

**The future interface must not decorate evidence posture. It must embody evidence posture as a governed conceptual system.**

## D. Interface Blueprint Definition

The Evidence Posture Workbench Interface Blueprint is a non-operational governance artifact that defines future interface zones, component roles, state displays, refusal displays, output envelope displays, copy boundaries, accessibility requirements, and performance constraints before any interface, prototype, engine, classifier, upload workflow, scoring system, form, API, or public route exists.

## O. Hoax Interface Thesis

The Evidence Posture Workbench interface must embody the core Hoax.ai idea:

**Evidence can look convincing before it is structurally usable.**

The interface must therefore make visible:

- the artifact before the subject;
- uncertainty before confidence;
- missing context before output;
- provenance gaps before conclusion;
- refusal before unsafe certainty;
- boundary before visibility;
- posture before verdict.

The interface must not be a dashboard of answers. It must be a chamber of constraints, states, and governed language.

Hoax.ai's interface should feel like entering a governed evidence chamber, not using a detector. The user should see that the system is not asking "fake or real?" but organizing the conditions that make evidence usable, uncertain, incomplete, boundary-limited, or not assessable.

## P. Hoax-Specific Interface Signature

The future interface must express a recognizable Hoax.ai signature through:

1. **Artifact-first framing** — The interface begins with the evidence artifact, not the person, institution, event, or claim attached to it.
2. **Boundary-first layout** — The artifact-subject boundary must be visible before any output envelope is visible.
3. **Posture-state architecture** — The interface must show evidence as moving through governed posture states, not as moving toward a score.
4. **Provenance shadow** — Missing origin, custody, source, or context must be visually treated as a shadow or absence that limits confidence.
5. **Not Assessable as a protected state** — Not Assessable must not appear as failure. It must appear as a disciplined protection against false certainty.
6. **Refusal as governance** — Refusal must not appear as an error. It must appear as a governed boundary action.
7. **Output envelope containment** — Any future output must look contained, limited, and bounded by labels. It must never look like a final judgment.
8. **No verdict gravity** — The interface must avoid any visual hierarchy that makes the user expect a final answer, score, green/red result, pass/fail mark, fake/real state, or binary classification.

## Q. Conceptual Interface Metaphors

**Allowed:** evidence chamber, posture field, boundary rail, provenance shadow, source trace, uncertainty register, refusal gate, output envelope, not-assessable lock, verification path.

**Forbidden:** detector, scanner, truth meter, risk score, fake/real switch, upload machine, forensic game, lie detector, courtroom verdict, policing dashboard, social media moderation panel, SaaS analytics dashboard.

## S. Interface Originality Requirement

The future interface must be difficult to confuse with common AI detector pages. It must not center on upload, scan, score, detect, verify, classify, or decide. It must center on posture, boundary, provenance, source context, missing information, refusal, not-assessable state, and output limits.

The interface should make Hoax.ai's category language visible through structure, not decoration.
